## Auditoría #1 — Superficie SECURITY DEFINER (seguridad 🔴) ✅

**Método:** inventario de las 54 funciones de `public` (prosecdef + ACL + si son trigger) cruzado contra los `.rpc()` reales del frontend (grep en `src`) y los tools del bot.

**Migración aplicada:** `audit1_least_privilege_execute_grants`.

**Resultado (verificado):** funciones ejecutables por `anon` **24 → 3**; por `authenticated` **41 → 33**.

### Tabla función → rol permitido
| Grupo | Funciones | Acción | Rol final |
|---|---|---|---|
| RPCs frontend expuestas a anon | `get_visible_patient_ids/staff_ids`, `get_patient_stats`, `get_appointment_trend`, `get_patient_ltv`×2, `get_retention_rate`×2, `get_service_analytics`×2, `get_appointment_prediction`×2, `get_stats_dashboard`×2 | REVOKE anon+PUBLIC | authenticated |
| RPCs solo-bot | `create_appointment`, `get_available_slots`, `get_patient_appointments`, `get_patient_profile`, `reactivate_bot` | REVOKE anon+authenticated+PUBLIC | **service_role** |
| Funciones de trigger | `enforce_*_limit`×3, `finance_guard`, `handle_audit_log`, `offers_set_updated_at`, `prevent_mass_delete`, `trigger_add_notification`, `trigger_audit_log`, `trigger_set_updated_at`, `update_plans_updated_at`, `validate_appointment` | REVOKE PUBLIC+anon+authenticated | solo owner (trigger) |
| Helpers de RLS (se dejan) | `get_user_business_id`, `is_business_active`, `has_feature`, `user_has_permission`, `get_effective_limit` | sin cambio | anon/auth (necesario para evaluar RLS; solo devuelven datos del propio caller) |
| Mantenimiento (ya locked) | `record_usage`, `check_rate_limit`, `create_monthly_partition`, `ensure_future_partitions`, `drop_old_partitions`, `apply_history_retention`, `run_dunning`, `record_payment` | — | service_role |

**Gap menor pendiente (documentado):** `get_visible_patient_ids/staff_ids` reciben `p_business_id` por parámetro y **no verifican** que el caller sea dueño de ese negocio → un authenticated podría pasar otro `business_id` y obtener UUIDs de pacientes ajenos (solo UUIDs opacos; para usarlos igual necesita RLS de `patients`). **Hardening recomendado:** dentro de la función, forzar `p_business_id := get_user_business_id()`. → Backlog P1.

## Auditoría #2 — Flujo de cobro / Dunning (producto 🔴) ✅

**Método:** trazado de `plan_status`/`plan_expires_at` desde el pago hasta el bloqueo (front `AccountStatusModal`, bot `Flow - ¿Plan Activo?`, RLS). Antes: la UI de suspendido/cancelado existía pero **nada** en la DB movía estados ni bloqueaba escritura; `is_business_active()` estaba huérfana.

**Migraciones aplicadas:** `audit2a_dunning_foundation` + `audit2b_rls_suspend_writes`.

**Implementado y verificado:**
1. Tabla `payments` (RLS on, solo service_role) — base para registrar pagos (y enchufar Stripe).
2. `record_payment(business_id, amount, method, note)` — inserta pago, **extiende `plan_expires_at` +1 mes**, pone `plan_status='active'`, quita `ai_paused`. Solo service_role.
3. `is_business_active()` ampliada a `plan_status IN ('active','trial')` (antes solo `active` → un trial quedaba bloqueado).
4. `run_dunning()` + cron **`run-dunning` (diario 08:00)**: `active`→`suspended` a los 7 días de vencido; `suspended`→`cancelled` a los 30.
5. **Suspensión con dientes**: `AND (SELECT is_business_active())` añadido a las políticas de **escritura** de 9 tablas de producto (appointments, patients, patient_phones, services, offers, income/expense_entries, supplies, service_supplies).

**Verificación (impersonando al usuario authenticated real, con rollback):**
`active_escribe: true` · `suspended_bloqueado: true` → un negocio suspendido queda **solo-lectura en el dashboard**; el bot (service_role) no se afecta y conserva su gate `¿Plan Activo?`.

**Pendiente (frontend, [TÚ]):** botón "Marcar pagado" en AdminPanel → `admin-update-business` que llame `record_payment`. El resto del ciclo ya es automático.

## Auditoría #3 — Rendimiento / cobertura de índices (rendimiento 🟠) 📋

**Método:** verificación estructural de que cada query caliente tiene un índice que la cubre por prefijo y orden. (La prueba de carga con 5×2,000 filas requiere una **branch de Supabase** — tiene costo por día; queda cost-gated para cuando el volumen lo justifique.)

### Matriz query → índice → veredicto
| Query caliente | Filtro/orden | Índice | Veredicto |
|---|---|---|---|
| Calendario | `business_id = X AND date_start ∈ [a,b)` | `idx_appt_business_date (business_id, date_start DESC)` | ✅ cubierta por prefijo+orden |
| Anti-doble-booking | overlap por rango | `appt_no_overlap` GIST (business_id, tstzrange) parcial | ✅ |
| Conversaciones | `business_id=X AND patient_id=Y ORDER BY created_at DESC` | `idx_history_biz_patient_created (business_id, patient_id, created_at DESC)` | ✅ calce exacto |
| Actividad (audit) | `business_id=X ORDER BY created_at DESC` | `idx_audit_biz_created (business_id, created_at DESC)` | ✅ |
| Lista pacientes | `business_id=X AND deleted_at IS NULL ORDER BY created_at DESC` | `idx_patients_business_active_created` parcial | ✅ |
| Búsqueda paciente | `display_name ILIKE / trigram` | `idx_patients_name_trgm` GIN | ✅ |
| Finanzas | `business_id=X ORDER BY occurred_at DESC` | `idx_income/expense_business_occurred` | ✅ |

**Veredicto:** cobertura completa de las queries calientes; no faltan índices para el modelo de acceso actual. **Recomendación:** cuando haya ≥3-5 negocios con miles de turnos, correr `EXPLAIN ANALYZE` en una branch para confirmar que el planner elige estos índices con datos reales (hoy, con tablas casi vacías, elegiría seq scan, lo cual es correcto). → Backlog P2.

## Auditoría #4 — Resiliencia del bot n8n (resiliencia 🟠) 📋

**Método:** revisión nodo por nodo de `onError`/`continueOnFail`/`retryOnFail` en el workflow.

**Ya resilientes ✅:** clasificadores y agentes (`retryOnFail` + `continueErrorOutput`), `Buffer - Crear` (continueOnFail), todos los `Historial - *` de handoff (`continueRegularOutput`).

**Nodos frágiles (sin manejo de error) — cortan la conversación en silencio si Supabase/Meta responden 4xx/5xx:**
| Nodo | Riesgo si falla |
|---|---|
| `DB - Request API Limit` | 🔴 hoy roto (key anon) + sin onError → corta todo mensaje. *(fix en doc 05)* |
| `Negocio - Obtener` / `Plan - Obtener` | Sin negocio/plan → todo el flujo muere sin responder |
| `Buffer - Obtener` / `Buffer - Eliminar` | Buffer no resuelto → sin respuesta |
| `Paciente - Crear` / `Crear Teléfono` | Un hipo de DB con contacto nuevo → conversación rota |
| `WA - Respuesta Basic/Pro/Enterprise` | Meta 4xx (p. ej. fuera de ventana 24h) → respuesta no entregada, sin log ni reintento |
| `Handoff - Activar *` | WA ya envió el mensaje pero `human_takeover` no se setea → estado inconsistente |
| `Notif - *` | Notificación perdida silenciosamente |

**Patrón de manejo recomendado (entregable):**
1. En los nodos de datos críticos (`Negocio/Plan/Buffer/Paciente`): `onError: continueRegularOutput` + rama que envía un WA amable ("Estamos con un problema técnico, intento de nuevo en un momento 🙏") y registra en `notifications` (tipo `error_ia`).
2. En los `WA - Respuesta`: `retryOnFail` (2 intentos) + `onError` que registre el fallo de entrega.
3. **Error Workflow global de n8n**: un workflow con *Error Trigger* que loguee cualquier excepción no capturada a una tabla `bot_errors` + notifique. Cierra el punto de "los errores mueren en silencio" del scorecard.

*(Solo documentación — las ediciones son en la UI de n8n, [TÚ].)*

## Auditoría #5 — Consistencia límites bot↔dashboard (producto 🟡) ✅

**Método:** matriz plan × límite × comportamiento, verificada con probes SQL (rollback).

| Recurso | Básico | Pro | Enterprise | Dashboard (authenticated) | Bot (service_role) |
|---|---|---|---|---|---|
| Pacientes | 50 | 150 | ∞ | **bloqueo duro** al llegar (trigger) | **nunca bloquea** (crea libre); visibilidad recorta a N (`get_visible_patient_ids`) |
| Staff | 1 | 5 | ∞ | bloqueo duro | n/a (el bot no crea staff) |
| Turnos/mes | 100 | 500 | ∞ | bloqueo duro | gate amable propio (`¿Superó límite?`) → mensaje + manual |
| Mensajes/mes | 1,000 | 5,000 | 20,000 | — | gate propio (`¿Conversaciones Ok?`) → mensaje |

**Verificado:** bot crea paciente #51 en Básico ✓; dashboard bloqueado en #51 ✓; `get_visible_patient_ids` devuelve exactamente 50 ✓ (probes anteriores). Coincide con el modelo: el límite es de **visualización** para el bot, **duro** para el dashboard.

**Inconsistencias detectadas (→ backlog):**
1. El bot **ignora `limit_overrides`** (lee `Plan - Obtener` crudo; el dashboard usa `get_effective_limit`). Un override de un negocio no lo respeta el bot. *(doc 05 #7)*
2. `Conversaciones - Contar` cuenta filas de `history` (user+assistant ≈ 2×) contra `max_conversations` → el límite efectivo es ~la mitad. *(doc 05 #5)*
3. Gate de turnos del bot usa `created_at` del mes; el trigger usa `date_start`. *(doc 05 #6)*
